# Blog Writing Guide

이 문서는 GitHub Pages 블로그 글을 작성할 때 사용하는 기준입니다.

## 1. 블로그의 역할

이 블로그는 다음 세 가지 역할을 동시에 수행합니다.

1. 실무 지식 저장소
2. 이력서 보조 자료
3. 포트폴리오 근거 자료

따라서 글은 단순 회고가 아니라, 읽는 사람이 작성자의 문제 해결 방식을 이해할 수 있도록 작성합니다.

## 2. 글 작성 기준

좋은 글은 다음 질문에 답합니다.

- 어떤 문제가 있었는가?
- 왜 문제가 발생했는가?
- 어떤 근거로 원인을 좁혔는가?
- 어떻게 해결했는가?
- 같은 문제가 다시 발생하면 어떻게 대응할 수 있는가?
- 이 경험이 어떤 개발자 역량을 보여주는가?

## 3. 기본 구조

장애 대응, 운영, 개발 문제 해결 글은 아래 구조를 사용합니다.

```markdown
## 문제점

## 원인

## 해결

## 실행 방법

## 검증 방법

## 재발 방지 / 개선 방향

## 포트폴리오 관점의 의미
```

## 4. 학습/개념 정리 구조

자격증, 보안/감사, 데이터베이스, 개발 개념 글은 아래 구조를 사용합니다.

```markdown
## 배경

## 핵심 개념

## 실무 적용 기준

## 예시

## 주의점

## 정리
```

## 5. 프로젝트 경험 일반화 기준

업무 경험을 공개 글로 바꿀 때는 반드시 일반화합니다.

| 원본 업무 기록 | 공개 글 표현 |
|---|---|
| 특정 고객사 서버 장애 | 공공기관 Linux 서버 장애 대응 |
| 내부 IP와 계정 포함 작업 | Linux 서버 접근 및 점검 절차 |
| 고객 회의록 | 요구사항 정리 과정에서 확인한 일반적 이슈 |
| 특정 DB명/스키마/테이블명 | 운영 DB 또는 분석 DB |
| 고객사 사업명 | 데이터 분석 시스템화 프로젝트 |

## 6. 제목 작성 기준

좋은 제목은 기술과 문제를 함께 드러냅니다.

좋은 예:

- Rocky Linux 디스크 마운트 실패 대응 절차
- Apache VirtualHost 404 장애 원인 분석
- Podman으로 CloudBeaver를 운영할 때 고려할 점
- 분석 결과를 웹 서비스로 시스템화할 때 필요한 기준

나쁜 예:

- 오늘 한 일
- 서버 문제 해결
- GitHub 블로그 시작
- 장애 정리

## 7. 민감정보 제거 체크리스트

공개 전 반드시 확인합니다.

- [ ] 고객사명 제거 또는 일반화
- [ ] 내부 IP 제거
- [ ] 계정명 제거
- [ ] 비밀번호/토큰/키 제거
- [ ] 내부 경로 제거 또는 일반화
- [ ] 회의록 원문 제거
- [ ] 계약상 비공개 내용 제거
- [ ] 개인정보 제거
- [ ] 보안상 악용 가능한 상세값 제거

## 8. 이력서 연결 기준

글은 이력서의 한 줄을 보완해야 합니다.

예시:

이력서 문장:

```text
Linux 기반 웹 서비스 운영 및 장애 대응 경험
```

연결할 블로그 글:

```text
Apache VirtualHost 404 장애 원인 분석
Rocky Linux 디스크 마운트 실패 대응 절차
```

## 9. 우선 작성할 글

현재 우선순위는 다음입니다.

1. ChatGPT를 활용해 GitHub Pages 기술 블로그를 시작한 이유
2. Rocky Linux 디스크 마운트 실패 대응 절차
3. Apache VirtualHost 404 장애 원인 분석
4. Podman 기반 CloudBeaver 운영 검토
5. Spring Boot + JSP 운영 환경 장애 대응
6. 분석 결과를 웹 서비스로 시스템화할 때 고려할 점

## 10. 글 품질 기준

글 하나는 아래 기준을 만족해야 합니다.

- 제목만 봐도 다루는 기술과 문제가 보인다.
- 문제 상황이 구체적이다.
- 원인 분석 과정이 있다.
- 실행 방법이 있다.
- 민감정보가 제거되어 있다.
- 이력서 또는 포트폴리오에 연결할 수 있다.
