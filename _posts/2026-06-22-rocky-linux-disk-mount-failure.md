## 문제점

Linux 서버를 재부팅한 뒤 기존에 자동으로 마운트되던 디스크가 정상적으로 마운트되지 않는 상황이 발생할 수 있습니다.

대표적인 증상은 다음과 같습니다.

- `df -h`에서 기존 마운트 경로가 보이지 않습니다.
- `lsblk`에서는 디스크나 파티션이 보이지만 `MOUNTPOINT`가 비어 있습니다.
- 서비스가 특정 데이터 경로를 찾지 못합니다.
- `/etc/fstab`에 등록된 항목 때문에 부팅이 지연되거나 emergency mode로 진입할 수 있습니다.

이 문제는 단순히 `mount` 명령만 다시 실행해서 끝낼 일이 아닙니다. 디스크 인식, 파일시스템 상태, UUID 변경 여부, `/etc/fstab` 설정 오류를 순서대로 확인해야 합니다.

## 원인

디스크 마운트 실패의 원인은 보통 다음 범위에 있습니다.

1. 디스크 또는 파티션이 OS에서 인식되지 않음
2. 파일시스템 손상
3. `/etc/fstab`에 잘못된 장치명 또는 UUID가 등록됨
4. 기존 `/dev/sdb2` 같은 장치명이 재부팅 후 바뀜
5. 마운트 대상 디렉터리가 없거나 권한이 맞지 않음
6. 이전에 수동 마운트만 했고 자동 마운트 설정이 없었음

특히 `/dev/sdb2` 같은 장치명은 고정값으로 보기 어렵습니다. 재부팅 후 디스크 탐지 순서가 바뀌면 다른 이름으로 잡힐 수 있습니다. 운영 서버에서는 장치명보다 UUID 기준으로 `/etc/fstab`을 작성하는 편이 안전합니다.

## 해결

대응 원칙은 다음입니다.

```text
디스크 인식 확인
-> 파일시스템 확인
-> UUID 확인
-> 수동 마운트 테스트
-> /etc/fstab 수정
-> 재부팅 검증
```

장애 상황에서는 바로 `/etc/fstab`을 수정하기보다 먼저 수동 마운트로 정상 접근 가능한지 확인해야 합니다.

## 실행 방법

### 1. 디스크와 파티션 확인

```bash
lsblk -f
blkid
df -h
```

확인할 항목은 다음입니다.

- 디스크가 보이는지
- 파티션이 보이는지
- 파일시스템 타입이 보이는지
- UUID가 있는지
- 기존 마운트 경로가 비어 있는지

예시:

```text
NAME   FSTYPE UUID                                 MOUNTPOINT
sdb
└─sdb2 xfs    11111111-2222-3333-4444-555555555555
```

### 2. 기존 fstab 확인

```bash
cat /etc/fstab
```

`/dev/sdb2`처럼 장치명으로 등록되어 있다면 UUID 방식으로 바꾸는 것이 좋습니다.

나쁜 예:

```fstab
/dev/sdb2 /data xfs defaults 0 0
```

좋은 예:

```fstab
UUID=11111111-2222-3333-4444-555555555555 /data xfs defaults,nofail 0 0
```

`nofail` 옵션을 넣으면 해당 디스크 마운트에 실패해도 부팅 전체가 중단되는 위험을 줄일 수 있습니다. 단, 서비스가 해당 경로에 의존한다면 별도 점검 로직은 필요합니다.

### 3. 마운트 대상 디렉터리 확인

```bash
ls -ld /data
```

없으면 생성합니다.

```bash
sudo mkdir -p /data
```

### 4. 수동 마운트 테스트

파일시스템 타입이 `xfs`인 경우:

```bash
sudo mount -t xfs UUID=11111111-2222-3333-4444-555555555555 /data
```

`ext4`인 경우:

```bash
sudo mount -t ext4 UUID=11111111-2222-3333-4444-555555555555 /data
```

마운트 후 확인합니다.

```bash
df -h
ls -la /data
```

### 5. fstab 적용 테스트

`/etc/fstab` 수정 후 바로 재부팅하지 말고 다음 명령으로 검증합니다.

```bash
sudo mount -a
systemctl daemon-reload
```

오류가 있으면 즉시 수정합니다.

```bash
findmnt --verify
```

### 6. 재부팅 검증

```bash
sudo reboot
```

재부팅 후 확인합니다.

```bash
lsblk -f
df -h
findmnt /data
```

## 검증 방법

정상 상태는 다음 조건을 만족해야 합니다.

- `lsblk -f`에서 대상 파티션의 `MOUNTPOINT`가 표시됩니다.
- `df -h`에서 마운트 경로가 표시됩니다.
- `findmnt /data`가 정상 결과를 반환합니다.
- 재부팅 후에도 동일하게 마운트됩니다.
- 해당 경로를 사용하는 서비스가 정상 기동됩니다.

## 재발 방지 / 개선 방향

운영 서버에서는 다음 기준을 적용하는 것이 좋습니다.

1. `/etc/fstab`에는 `/dev/sdb2` 대신 UUID를 사용합니다.
2. 데이터 디스크에는 `nofail` 옵션 적용을 검토합니다.
3. 마운트 경로를 사용하는 서비스는 `RequiresMountsFor=` 같은 systemd 의존성 설정을 검토합니다.
4. 디스크 증설, 교체, 파티션 변경 후에는 재부팅 검증까지 수행합니다.
5. 장애 대응 기록에는 `lsblk -f`, `blkid`, `/etc/fstab`, `df -h` 결과를 함께 남깁니다.

## 포트폴리오 관점의 의미

이 사례는 단순 Linux 명령어 사용 경험이 아니라 운영 서버에서 장애 원인을 단계적으로 좁히는 역량을 보여줍니다.

특히 다음 역량과 연결됩니다.

- Linux 파일시스템과 마운트 구조 이해
- 재부팅 후 자동 복구 가능한 운영 설정
- `/etc/fstab` 변경 시 부팅 장애 리스크 인지
- 수동 조치와 영구 설정을 분리하는 운영 습관
- 장애 대응 결과를 재사용 가능한 체크리스트로 정리하는 능력
